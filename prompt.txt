

Build a production-ready **Bulk SMS Web Platform (obubaka.com-style)** using **Core PHP (no frameworks)** and **MySQL**, designed as a reseller-style SaaS where clients create accounts, log in, buy SMS credits, manage contacts, and send both normal bulk SMS and personalized SMS. The platform must be clean, fast, secure, and use a modular file structure. The system must use a **dynamic BASE_URL** from the start (auto-detect protocol + domain + subfolder) so the same code works on localhost, in a subfolder, and on a live domain without editing links. All internal links, assets, redirects, and API endpoints must use BASE_URL (never hardcode URLs).

The system must support **account branding per user** where each client sets a business name and uploads a business icon/logo used in the sidebar/header and across exports. Each user must have a complete profile (name, phone, email, address) and be able to update it. The platform must include a **user activation/deactivation system** controlled by admin: users can be set to active/inactive/suspended, and inactive users must be blocked from login and sending; if a user is disabled while logged in, the system must force logout and show a clear message. Also include a **maintenance mode** feature controlled by admin via settings; when maintenance is enabled, all clients are redirected to a maintenance page showing the maintenance message, while admin accounts remain allowed to log in and access the dashboard. Maintenance enforcement must happen globally via bootstrap/middleware, not only on login.

The UI must match a modern SaaS dashboard with a **collapsible sidebar**. The sidebar behavior must be exactly: when collapsed, it shows only icons and hovering over a category icon displays its submenu in a floating flyout; when expanded, categories must be clicked to expand/collapse submenus. The active page must keep its submenu open and highlighted automatically (server-side detection), so users always see where they are. The dashboard must display current SMS credit balance prominently and provide quick access to Compose SMS, Personalized SMS, Sent Messages, Phonebook, Wallet/Top Up, Me2U transfer, Notifications, Inbox, Settings, and Logout. Admin users must see additional admin menus.

Implement a full **theme system** with up to **10 selectable themes including a professional dark theme**. Themes must be implemented using a CSS variable system so the UI changes cleanly without breaking components. The user must be able to choose a theme from a Theme Settings page, preview it, save it to their profile, and have it apply immediately (update session), and the theme must persist across sessions. The system must load theme styles dynamically from `assets/css/themes/{theme_key}.css`.

Core messaging functionality must include **Bulk SMS** and **Personalized SMS**. Bulk SMS must allow manual input of recipient numbers, supporting comma-separated and newline-separated formats, accepting Uganda formats like `2567xxxxxxx`, `+2567xxxxxxx`, `077xxxxxxx`, and must normalize numbers to E.164 (+256...) or a consistent stored format. The system must automatically remove duplicates and invalid numbers and show counts of valid/invalid/duplicates before charging. The system must calculate message length and SMS segments (GSM/Unicode basics can be approximated initially) and compute the total billable SMS units as recipients × segments. Sending must support Send Now and Schedule Later with a date-time selector. Scheduled messages must be stored in a queue table and processed by a worker/cron script that safely locks jobs to avoid double sending.

Personalized SMS must support uploading an Excel `.xls` file containing recipients and variables. The system must provide a downloadable sample template. The message body must support placeholders like `@@name@@`, `@@amount@@`, and `@@var1@@` through `@@var5@@`, replaced per recipient row. The system must store per-recipient variables JSON and generate the final message per recipient when sending. Both sending modes must produce send logs with sender id, recipients, message preview, date/time, total SMS units, and status. Delivery responses must be tracked per recipient (queued/sent/failed) with gateway response IDs and error messages.

The platform must include a **Phonebook module**: users can create contact groups, add contacts (name + phone), bulk paste numbers, and add many numbers to a group at once. Each user’s contacts must be isolated (multi-tenant separation by user_id). A default “All” group should exist or be created automatically per account. Provide search, pagination, and safe duplicate prevention (unique per user per number).

Implement a **wallet/credits system** where each user has an SMS balance. Sending SMS must debit credits atomically and prevent sending if balance is insufficient. Credits are added through Top Up and Vouchers. The Buy/Top Up page must allow entering amount (minimum 15,000 UGX) and payer phone number, and record a topup request with pending/approved/failed status. Admin must have a payment approval interface to approve topups, which credits the user wallet and records a wallet transaction audit. Implement **Me2U** credit transfer where a user can transfer SMS units to another user by username; record both transfer logs and wallet transactions (transfer_out and transfer_in). Implement voucher loading where users enter a voucher code, the system validates it (unused/not expired), credits the account, and marks the voucher as used with timestamps.

Add an **internal messaging system** (support inbox) where users can message admin/support. Build it as thread-based tickets with subject, category (billing/sms/sender_id/technical/general), priority, and status (open/pending/resolved/closed). Users can view inbox, open a thread, and reply. Admin can view all tickets, respond, and optionally broadcast announcements to all users. Messages must support read/unread tracking.

Add a **push notifications module** using Web Push subscriptions (VAPID approach or equivalent) for browser notifications. Users can opt-in, and the system must store push subscriptions per user and device. The system must generate notifications for events such as topup approved, SMS sent/failed/low balance, scheduled campaign started/completed, and new internal message. The UI must include a Notifications page showing a notification center with read/unread states, mark-as-read, and a bell icon indicator.

Implement **admin panel features** including: user management (create/edit/activate/deactivate/suspend), pricing management per user per network, topup approval dashboard, global SMS logs, maintenance mode settings, voucher creation, broadcast messaging, and audit log viewing. The system must support **per-user pricing per network** (MTN/Airtel/Warid/UTL/Africell/Others) stored in a pricing table; cost calculation can use user-specific price per SMS or points per SMS, but must be consistent with the wallet debit logic.

Security must be taken seriously: use prepared statements everywhere (MySQLi), session hardening, CSRF tokens on all forms, server-side validation for phone numbers and file uploads, safe file upload handling for logos, and strict authorization checks (role-based access control). Include an `audit_logs` table and log major actions like login, sending SMS, approvals, credit transfers, user activation changes, theme changes, and maintenance changes. Add pagination/search to large tables like sent messages, contacts, and transactions. Implement exports (Excel/CSV at minimum) for sent logs and transactions.

Deliver the complete codebase in the specified modular structure with root files (`index.php`, `login.php`, `logout.php`, `dashboard.php`, `maintenance.php`), shared includes (`config.php` with dynamic BASE_URL, `bootstrap.php`, `db.php`, `auth.php`, `csrf.php`, `helpers.php`), templates (`header.php`, `sidebar.php`, `footer.php`), assets (`app.css`, `app.js`, theme CSS files, service worker `sw.js`), and modules folder structure for SMS, phonebook, wallet, messaging, notifications, settings, and admin. Provide a complete `schema.sql`, `seed.sql`, and a `sample_accounts.php` script that creates demo accounts using proper `password_hash()`. Ensure all pages use BASE_URL for links and that the sidebar behavior and theme system work exactly as specified.
